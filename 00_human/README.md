# 00_human（人工区 · 最终定版）

本目录是 **项目中唯一需要人工参与与确认的区域**。
其结构与职责已定版，后续项目中 **不应新增文件类型**，仅允许更新既有文件内容。

本区的定位是：**盖章 / 决策 / 制片信息面板**  
而不是讨论区、草稿区或生产区。

---

## 一、人工在本系统中的唯一职责

人工只做三件事：

1. **确认阶段性产物（Approval）**
2. **做少量方向性决策（Decision）**
3. **提供或确认制片/管理信息（Production）**

除此之外的所有生成、拆解、推导工作，均由 Codex 自动完成。

---

## 二、文件职责说明（固定，不可扩展）

### 1. INBOX.md —— 全流程 gate 看板（All Gates）

- 显示 `PIPELINE_STAGES.yaml` 定义的所有 gate，按 stage_order 排序
- Status 会标记为 `approved` / `blocked` / `pending`
- 每行同时告诉你 artifact 是否存在以及哪些 approvals 仍未达成

**把整个节奏摆在 INBOX 中；深度追踪流程进度时请直接打开本表。**

### 2. NOW.md —— 当前可行动的 pending gate

- 仅显示 Status 为 `pending`，且 prerequisites 已满足、artifact 已存在的 gate
- 表格按相同 stage_order 排列，保持“今天该做什么”的聚焦视图
- 该文件会随 `preflight` 每次刷新：不 action 的 gate 将自动消失直到满足条件

**INBOX 展示全流程状态，NOW 则告诉你“现在可以干啥”。**

---

### 2. APPROVALS.md —— 不可逆确认记录（锁死）

- 记录你已确认的阶段性产物
- 一旦写入，表示：
  - 对应文件内容被 **锁定**
  - Codex 不得覆写，只能生成新版本（v2、v3…）

**Approval 等同于签字确认，是流程继续的硬条件。**

---

### 3. DECISIONS.md —— 方向性决策记录（低频）

- 记录创作或制作方向上的选择
- 例如：是否旁白、风格取向、节奏快慢、是否多版本等
- 不记录执行细节

**Decision 一旦记录，将对下游产生约束，不应反复修改。**

---

### 4. NEEDED_INPUTS.md —— 阻塞说明（临时）

- 仅在项目无法继续推进时使用
- 用于列出“缺什么 / 放在哪”
- 输入补齐后，该文件应恢复为空状态

**这是“红灯提示”，不是待办清单。**

---

### 5. PRODUCTION.md —— 制片/管理信息（可选但推荐）

- 记录制片视角的事实与约束
- 如：目标时长、交付规格、多版本要求、语言版本等
- 不参与流程判断，不作为执行闸门

**这是给人使用的管理备忘，不是给 Codex 的逻辑输入。**

---

## 三、重要行为规则（必须遵守）

- INBOX 中已完成的事项 **不会再次出现**
- 所有历史确认应进入 APPROVALS / DECISIONS
- 不在本目录新增其他类型文件
- 不在 docs/ 或 inputs/ 中进行人工确认行为

---

## 四、设计原则（一句话）

> **如果一条信息不是必须由人判断的，它就不应该出现在 00_human。**

本目录的结构与语义已冻结。
后续项目中，仅更新内容，不再调整结构。

---

## 模式开关（Template vs. Project）

- 本模板默认 `30_project/inputs/index.yaml` 中 `project_mode: template`。
- 模板模式下 `NOW.md` 只给出提示，`INBOX.md` 仅展示 gate 状态而不会自动推进。
- 复制本仓库进入新项目后，务必将 `project_mode` 改为 `project` 并补充真正的素材，才能看到 actionable gate 并进入生产节奏。
